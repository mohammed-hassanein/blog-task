01. الملخص التنفيذي Executive Summary

تهدف المنصة إلى توحيد العمليات اليومية لشركات السياحة والسفر داخل نظام SaaS مركزي يسهّل إدارة البيانات والتشغيل من واجهة واحدة.

تغطي المنصة إدارة البرامج السياحية، الحجوزات، العملاء، الفواتير، المدفوعات، المصروفات، والتقارير التشغيلية والمالية، بهدف تقليل الاعتماد على الجداول اليدوية والملفات المتفرقة.

يعتمد نموذج التشغيل على اشتراكات دورية، مع دعم وحدات إضافية قابلة للتفعيل لكل شركة حسب احتياجاتها التشغيلية.

تتضمن المنصة بوابة حجز عامة لكل شركة عبر نطاق فرعي خاص، تتيح للعملاء تصفح البرامج السياحية وتقديم طلبات الحجز عبر الإنترنت.

الخصائص الرئيسية

نموذج متعدد الشركات (Multi-Company)بنية تعتمد على قاعدة بيانات مشتركة مع عزل تام وشامل للبيانات بين الشركات، مع تحديد هوية كل شركة عبر نطاق فرعي خاص بها.
تصميم مودلر مرن (Modular Design)إمكانية تفعيل أو تعطيل الوحدات والخصائص لكل شركة بشكل مستقل ومباشر وفقاً لنوع اشتراكها.
منصة ثنائية اللغة بالكاملدعم كامل وأصيل للغتين العربية والإنجليزية على مستوى النظام بالكامل، مع اتجاه واجهة ديناميكي يتبدل تلقائياً حسب اللغة المختارة.
معمارية جاهزة للتوسع بالقوالبفصل واضح بين منطق العمل وطبقة العرض، مع بدء المرحلة الأولى بقالب واحد موحد وإمكانية إضافة قوالب جديدة لاحقاً عبر Theme Gallery.

تقارير تشغيلية فوريةرؤية لحظية ومباشرة لحالة الحجوزات، الإيرادات المحصلة، المصروفات، وأداء الموظفين العام.
قابلية عالية للتوسع (Scalable)بنية تقنية مرنة ومدروسة تسمح بإضافة وحدات وخدمات مستقبلية متطورة دون إعادة بناء النظام من الصفر.

  • توحيد إدارة البرامج السياحية، الحجوزات، بيانات العملاء، والحركات المالية ضمن منصة مركزية متكاملة واحدة.
  • توفير تقارير تشغيلية وإدارية دقيقة ومباشرة تدعم صناع القرار في الوقت الفعلي وببيانات موثوقة.
  • بناء نظام مرن وقابل للتوسع المستقبلي لاستيعاب خدمات تكنولوجية جديدة وإضافات متطورة دون قيود فنية.
  • توفير أساس تقني موحد يمكن لشركات السياحة الاعتماد عليه في رقمنة عملياتها وتوسيع خدماتها.

03. المستخدمون والأدوار Roles & Users

يتضمن النظام هيكلة صلاحيات دقيقة ومقسمة على مستويين رئيسيين لضمان التحكم الكامل، وحماية البيانات، وسهولة التشغيل الإداري. يتم تطبيق هذه الهيكلة فنياً عبر Spatie Laravel Permission بحيث يكون كل دور مرتبطاً بالشركة التابعة له.

مستوى المنصة الشامل (Platform Level)

الدور الوظيفيالمهام والمسؤوليات الواسعة

04. المعمارية التقنية Technical Architecture

تعتمد المعمارية الفنية على بنية Laravel MVC تقليدية مع العرض من جهة الخادم، مع التركيز على الاعتمادية، الأمان، وسهولة التوسع.

4.1 حزمة التقنيات المستخدمة (Technology Stack)

الطبقةالتقنية المعتمدةالغرض البرمجي والوظيفة الفنية
BackendLaravel 13 (PHP 8.4)بناء منطق العمل الأساسي للنظام، معالجة البيانات، وإدارة العمليات التشغيلية ضمن MVC.
FrontendLaravel Blade + Bootstrap 5واجهات مستخدم تفاعلية وسريعة الاستجابة عبر Server-Side Rendering دون اعتماد نمط SPA.
DatabaseMySQL 8تخزين وإدارة البيانات المهيكلة وعلاقات النظام.
CacheRedisإدارة الجلسات وتخزين البيانات المؤقتة لرفع سرعة النظام.
QueueLaravel Horizonإدارة ومعالجة الوظائف الخلفية والمهام المجدولة والتلقائية.
Cloud StorageAWS S3 / Cloudflare R2حفظ الملفات والفواتير والمستندات والصور المرفوعة بشكل سحابي آمن.
AuthenticationLaravel Session Auth + RBACتأمين تسجيل الدخول والجلسات عبر Session-Based Authentication والتحقق من الأدوار والصلاحيات.
SearchLaravel Scoutمحرك بحث متطور يتيح البحث السريع في بيانات النظام.

4.2 معمارية العزل بين الشركات (Multi-Company Architecture)

يعتمد النظام على قاعدة بيانات مشتركة مع عزل منطقي للبيانات على مستوى Query Scope، وتحديد سياق الشركة من خلال النطاق الفرعي.

Company Identificationتحتوي كل شركة على معرف company_id يتم إدراجه تلقائياً في كافة جداول البيانات ويرتبط مباشرة بالنطاق الفرعي الخاص بالشركة.
Data Isolationتطبيق Global Query Scope يمنع أي شركة من استعراض أو الوصول لبيانات شركة أخرى.
Company Resolutionيتم التعرف على هوية الشركة تلقائياً عبر النطاق الفرعي مثل company-a.platform.com دون الاعتماد على مسارات من نوع platform.com/company-a.
Middleware Contextيقوم CompanyMiddleware بفحص النطاق الفرعي وتحديد سياق الشركة والبيانات والقالب النشط واللغة الافتراضية.
Scalabilityتسمح البنية مستقبلاً بترقية الشركات الكبيرة جداً إلى قواعد بيانات منفصلة تماماً.

4.3 معمارية الوحدات والبرمجيات (Module Architecture)

  • الاستقلالية التامة: تعمل كل وحدة برمجية بشكل منفصل، مما يتيح تفعيلها أو تعطيلها دون التأثير على عمل النظام الأساسي.
  • سجل الوحدات المركزي: لوحة تحكم مركزية تتابع حالة تفعيل موديلات النظام لكل شركة، وتشمل القالب النشط لكل شركة كأحد عناصرها القابلة للتفعيل والتبديل.
  • واجهة موحدة: توفير واجهات API مستقرة وموحدة لتسهيل تناقل البيانات بكفاءة بين الموديلات المختلفة.
  • التحكم القائم على الأدوار: نظام صلاحيات دقيق ومفصل يحدد تصرفات وأدوار المستخدمين داخل كل وحدة مستقلة عبر Middleware وSpatie Laravel Permission.

4.4 الأمان ومكافحة المخاطر (Security Measures)

  • إدارة وتأمين جلسات المستخدمين عبر Session-Based Authentication والتكامل مع Redis.
  • تفعيل حماية CSRF على كافة النماذج البرمجية.
  • تطبيق Rate Limiting وRequest Throttling لحماية الخوادم من الإغراق وإساءة الاستخدام.
  • تنقية المدخلات عبر آليات Laravel الافتراضية لمنع SQL Injection وXSS.
  • تشفير البيانات الحساسة أثناء الحفظ وأثناء الانتقال عبر TLS.
  • إدارة الأدوار والصلاحيات عبر RBAC وSpatie Laravel Permission.
  • سجل نشاط شامل وموثق عبر Spatie Activity Log.

4.5 معمارية القوالب والعرض (Theming Architecture)

تعتمد المرحلة الأولى على قالب موحد واحد للبوابة العامة، مع تصميم البنية البرمجية وقاعدة البيانات بطريقة تسمح بإضافة معرض قوالب لاحقاً دون إعادة هيكلة الكود أو منطق العمل.

Theme Registryجدول themes يحتوي حالياً على سجل default ويستخدم مستقبلاً كقائمة مرجعية للقوالب الإضافية.
Active Themeعمود active_theme ضمن إعدادات كل شركة بقيمة افتراضية default.
View Namespacesتنظيم واجهات البوابة العامة داخل resources/views/themes/default/portal.
Brandingتخزين شعار الشركة والألوان والخطوط ونصوص التعريف بشكل مستقل عن القالب وحقنها كمتغيرات CSS.
Blade Componentsبناء صفحات البوابة من مكونات Blade مستقلة مثل بطاقة البرنامج، يوم من خط السير، نموذج الحجز، وملخص الدفع.

4.6 معمارية التعدد اللغوي والاتجاه (i18n & RTL/LTR Architecture)

يشمل دعم اللغتين العربية والإنجليزية واجهات النظام الثلاث: لوحة المنصة، لوحة الشركة الداخلية، والبوابة العامة للحجوزات.

الواجهةآلية تحديد اللغة
لوحة المنصةعمود preferred_locale في جدول users لكل مستخدم من فريق المنصة.
لوحة الشركة الداخليةpreferred_locale لكل موظف، مع استخدام default_locale للشركة كقيمة مبدئية عند أول تسجيل دخول.
البوابة العامة للحجوزاتdefault_locale لكل شركة مع إمكانية تبديل اللغة يدوياً وحفظ الاختيار عبر بادئة المسار.

تصنيف المحتوى القابل للترجمة

نوع المحتوىأمثلةآلية التخزين والترجمة
نصوص الواجهة الثابتةتسميات الحقول، الأزرار، رسائل النظام، حالات الحجز.ملفات Laravel القياسية lang/ar وlang/en.
محتوى تنشئه الشركةاسم البرنامج، الوصف، خط السير، الموارد، بنود يشمل/لا يشمل.أعمدة JSON قابلة للترجمة بصيغة {"ar":"...","en":"..."}.
بيانات العملاء الشخصيةاسم العميل، بيانات الاتصال، ملاحظات المسافرين.لا تترجم وتعرض كما أدخلها الموظف أو العميل.

تضاف بادئة اللغة إلى مسار روابط البوابة العامة فقط مثل company-a.platform.com/ar/programs/123 وcompany-a.platform.com/en/programs/123.

تستخدم Bootstrap 5 بنسخته الداعمة للاتجاه من اليمين لليسار، وتكتب أنماط CSS المخصصة باستخدام الخصائص المنطقية مثل margin-inline-start وpadding-inline-end.

يعرض التاريخ الهجري بجانب التاريخ الميلادي في كل مكان يظهر فيه تاريخ ضمن النظام، مع تخزين التاريخ ميلادياً دائماً وتحويله عند العرض فقط.

05. وحدات النظام الأساسية Core System Modules

5.1 إدارة علاقات العملاء (CRM)

تغطي وحدة العملاء بيانات العميل الأساسية، بيانات الاتصال، تفضيلات اللغة، ملاحظات المسافرين، والربط المباشر بالحجوزات والملفات التشغيلية.

5.2 الإدارة المالية

تدير الوحدة المالية عروض الأسعار، الفواتير، المدفوعات، والمصروفات التشغيلية المرتبطة بكل حجز وملف. وتولد جميع المستندات المالية الصادرة من هذه الوحدة بلغة العميل المسجلة في ملفه عبر مكتبة PDF تدعم تشكيل الحروف العربية بشكل سليم ومتصل.

5.3 إدارة الحجوزات (Booking Management)

وحدة ذكية لإدارة دورة حياة الحجز التشغيلية والملفات بالكامل، بدءاً من تقديم الطلب الأولي حتى إتمام الرحلة وإغلاق الملف الفعلي.

حالة الحجزالوصف التشغيلي للحالة في النظام
Pendingطلب حجز جديد وارد للنظام بانتظار المراجعة الفنية من موظفي المبيعات والتحقق من التفاصيل.
Approvedتمت مراجعة الطلب والموافقة عليه تشغيلياً وهو بانتظار تأكيد العميل النهائي وسداد الدفعة المالية.
Confirmedحجز مؤكد بالكامل، تم استلام الدفعات المرتبطة به، وأصبح الملف جاهزاً للتنفيذ الفعلي.
Cancelledحجز ألغي بناءً على طلب العميل أو الشركة مع توثيق أسباب الإلغاء والرسوم إن وجدت.
Completedتم تنفيذ الرحلة بنجاح وإغلاق ملف العميل وتحويله لتقييم تجربة الخدمة.

5.4 إدارة المستخدمين والصلاحيات

  • إدارة حسابات الموظفين وإضافة وتعديل بياناتهم أو تجميد حساباتهم التشغيلية عند الحاجة.
  • إدارة الأدوار والصلاحيات بشكل تفصيلي داخل كل وحدة عبر Middleware وSpatie Laravel Permission مع Teams.
  • توثيق زمني صارم للعمليات الحساسة عبر Spatie Activity Log.

5.5 التقارير التشغيلية والإدارية

التقريرالمحتوى والتحليلات المستهدفة
تقارير الحجوزاتإحصائيات الحجوزات المؤكدة والمعلقة والملغاة خلال أي فترة زمنية محددة.
تقارير العملاءتحليل سلوك العملاء، رصد العملاء الأكثر حجزاً، ومعدلات اكتساب العملاء الجدد.

06. البوابة العامة للحجوزات Public Booking Portal

توفر المنصة لكل شركة بوابة حجز عامة عبر نطاق فرعي خاص بها، تتيح لعملائها تصفح البرامج السياحية وتقديم طلبات الحجز مباشرة عبر الإنترنت. تعتبر البوابة العامة جزءاً أساسياً من النواة الجوهرية للمنصة.

  • تعمل البوابة ضمن سياق الشركة المحدد عبر النطاق الفرعي.
  • تستخدم القالب النشط للشركة والهوية البصرية المخزنة ضمن إعداداتها.
  • تعرض المحتوى باللغتين العربية والإنجليزية حسب اختيار الزائر أو اللغة الافتراضية للشركة.
  • تسمح بتصفح البرامج، عرض تفاصيل خط السير، وتقديم طلبات الحجز عبر الإنترنت.

07. البرامج السياحية Tour Programs

تم تصميم وحدة البرامج السياحية لتكون مرنة وقابلة للتخصيص بالكامل. البرنامج السياحي في هذا النظام مصمم كبرنامج خاص يطلبه العميل خصيصاً لعائلته أو مجموعته المستقلة، وبالتالي لا يفتح للحجز الجماعي العام أو الأفواج المشتركة.

5. نمط الحجز وإدارة المسافرين

  • ربط الإقامة: ربط الفنادق أو المنتجعات أو أماكن الإقامة بأيام الإقامة المحددة في خط السير.
  • عدد البالغين: قيمة مرجعية افتراضية قابلة للتعديل بالكامل عند بدء الحجز.
  • عدد الأطفال: الافتراضي في النظام هو صفر، مع إمكانية تحديد الأعمار عند إجراء الحجز.
  • ملاحظات المسافرين: توثيق الاحتياجات الخاصة مثل الغرف المتصلة أو تفضيلات الوجبات.

عند تحويل البرنامج إلى حجز فعلي، يقوم موظف المبيعات بمراجعة وتعديل أعداد المسافرين وتواريخهم الفعلية لتتوافق تماماً مع العائلة المستهدفة، مع إمكانية إعادة حساب التسعير آلياً بناءً على الأعداد الجديدة.

08. مكتبة الموارد Resource Library

مكتبة الموارد هي المساحة المركزية والموحدة داخل النظام لإدارة كافة العناصر والموردين الذين تتم الاستعانة بهم لبناء البرامج السياحية. تتيح إدخال بيانات الموارد مرة واحدة ثم إعادة استخدامها وربطها بآلاف البرامج المختلفة دون تكرار الإدخال اليدوي.

بدلاً من الاعتماد على كيان عام واحد resource، يبنى كل نوع من الموارد كنموذج Eloquent وجدول قاعدة بيانات مستقل بحقول ومتطلبات خاصة تناسب طبيعته. تشترك الكيانات في Traits جاهزة: BelongsToCompany، HasTranslations، وInteractsWithMedia.

الربط المرجعي المرن بين الموارد والأشكال (Polymorphic Relation)

يحتوي جدول بنود خط السير itinerary_items على عمودي resourceable_type وresourceable_id بحيث يمكن لأي بند الإشارة إلى سجل من أي كيان من الكيانات الستة دون ربط مباشر بجدول معين، مع السماح بإضافة أنواع موارد جديدة مستقبلاً دون تعديل بنية الجدول نفسه.

09. سير عمل الحجوزات Booking Workflows

يوضح هذا القسم الآلية التي يتحول بها البرنامج السياحي المصمم إلى حجز فعلي مرتبط بعميل محدد، بدءاً من أول استفسار وحتى إغلاق الملف بعد انتهاء الرحلة.

مراحل سير العمل الأساسي

  1. استقبال طلب العميل سواء عبر البوابة العامة أو مباشرة من خلال موظف المبيعات.
  2. تحويل البرنامج السياحي المختار إلى حجز مبدئي مع تحديد التواريخ الفعلية وعدد المسافرين البالغين والأطفال.
  3. إصدار عرض سعر أو فاتورة مبدئية للعميل توضح التكلفة الإجمالية بعد إعادة حساب الأعداد الفعلية.
  4. تأكيد الحجز رسمياً عند استلام دفعة العربون أو السداد الكامل وتسجيل المعاملة المالية.
  5. متابعة تنفيذ الرحلة على الأرض والتنسيق مع الموردين مثل الفنادق والنقل والمرشدين.
  6. إغلاق الحجز بعد انتهاء الرحلة وتوثيق ملاحظات العميل وتقييمه.

دورة حياة حالة الحجز (Booking Status Lifecycle)

الحالةالوصف التشغيلي
Inquiryطلب أولي من العميل لم يتحول بعد إلى حجز رسمي.
Draft Bookingتم تحويل البرنامج إلى حجز وربطه بملف العميل، لكن لم تستلم أي دفعة بعد.
Confirmed – Deposit Paidتم استلام دفعة جزئية وتأكيد الموارد المرتبطة لدى الموردين.
Fully Paidتم تحصيل كامل المستحقات المالية للحجز قبل بدء تنفيذ الرحلة.
In Progressالرحلة جارية فعلياً على الأرض وفريق التشغيل يتابع خط السير يوماً بيوم.
Completedانتهت الرحلة بنجاح وتم إغلاق الملف وتوثيق تقييم العميل.
Cancelledتم إلغاء الحجز قبل تنفيذه مع تسجيل سبب الإلغاء وأي مبالغ مستردة مرتبطة به.

يحمل كل حجز رقماً مرجعياً فريداً يستخدم في كافة المستندات المرتبطة به، وتعرض جميع التواريخ المرتبطة بالحجز بالتقويمين الهجري والميلادي معاً.

11. خارطة الطريق التطويرية Development Roadmap

تمت إعادة ترتيب أولويات خارطة الطريق اعتماداً على القرارات المعمارية الموضحة في قسم المعمارية، بحيث تصبح بنية التعدد اللغوي، بنية القوالب، والحزم البرمجية الجاهزة جزءاً أساسياً من مرحلة الأساس وليست بنوداً تضاف لاحقاً.

المرحلةالمحتوى الرئيسي والمخرجات
المرحلة 1: Foundationتجهيز Laravel 13، عزل الشركات عبر النطاقات الفرعية، تثبيت وتهيئة Spatie Permission مع Teams، Activity Log مع company_id، Media Library، إعادة بناء Breeze بBootstrap 5، بنية i18n، بنية القوالب، التقويم المزدوج، والمصادقة.

المرحلة 3: Public Booking Portalبناء واجهات البوابة العامة على القالب الافتراضي، تفعيل تبديل اللغة عبر /ar/ و/en/، تصفح البرامج، تقديم طلبات الحجز، والدفع عبر الإنترنت.
المرحلة 4: Reports & SaaS Layerالتقارير التشغيلية والمالية والإدارية، وطبقة إدارة المنصة بما فيها سجل تفعيل الوحدات والقوالب.

يعرض هذا المستند تصوراً فنياً لمنصة SaaS لإدارة شركات السياحة والسفر، يجمع بين العمليات التشغيلية والمالية ومعمارية قابلة للتوسع تشمل القوالب، التعدد اللغوي، دعم الاتجاهين، التقويم المزدوج، والحزم البرمجية المعتمدة.
